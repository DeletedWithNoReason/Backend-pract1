Даний проект реалізує кастомну систему автентифікації та управління даними, адаптовану під специфіку об’єктно-орієнтованого термінала. Нижче наведено ключові відмінності від стандартної архітектури Django.

1. Реєстрація операторів
Замість стандартної команди createsuperuser, у проекті реалізовано кастомну команду objreg

   Команда: python manage.py objreg

Параметри реєстрації:

  Employee ID (Username): Суворо 8 символів. Тільки цифри та велика латиниця (наприклад, DAVID198).

  Access Key (Password): Суворо 5 символів. Тільки цифри та велика латиниця (наприклад, AX402).

  First/Last Name: Текстові поля для відображення у звітах.

2. Протокол автентифікації
Система використовує кастомну модель користувача, що наслідується від AbstractUser, але переписує логіку перевірки:

  Валідація: При вході система виконує full_clean() з ігноруванням унікальності для перевірки відповідності регулярним виразам (Regex).

  Error Handling: Замість стандартних помилок Django ("Це поле обов'язкове" тощо), термінал видає універсальні коди помилок:

  PROTOCOL_VIOLATION: INVALID_DATA_STRUCTURE — невідповідність формату (довжина, символи).

  TERMINAL_ERROR: INVALID_CREDENTIALS — дані вірні за форматом, але оператора не існує в базі.

3. Управління сесіями
  Login: Реалізований як "вхідний шлюз". Доступ до будь-яких систем (/systems/) закритий для неавторизованих користувачів.

  Logout: Функція TERMINATE_SESSION повністю очищує сесійні куки та скидає статус оператора, повертаючи термінал у стан очікування.

  Identification: На сторінках списку систем відображається поточний активний оператор (user.first_name, user.last_name), що підтягується безпосередньо з активної сесії.
